<objective>
Turn the staging tables into the materialized album universe: join release-group identity to its primary artist and first-release year, gate on the ListenBrainz popularity floor so the universe is tens of thousands (not millions), write each surviving row into `entities` as `(entity_type='album', mbid)`, and make the whole operation idempotent via upsert on the composite PK so a refreshed dump never duplicates.

Purpose: This is the payload of the phase — the queryable universe the Phase 2 pick loop draws pairs from. The notability floor (DATA-04) is what keeps the pool honest, and the upsert (DATA-05) is what makes bi-weekly dump refreshes safe.
Output: `pipeline/materialize.py`, `pipeline/config.py`, `pipeline/test_materialize.py`, `pipeline/README.md` documenting the full end-to-end run.
</objective>

<notability_floor>
The floor is a tunable threshold on the ListenBrainz popularity signal (listener_count preferred; it is steadier than raw listen_count). Put it in pipeline/config.py as `NOTABILITY_MIN_LISTENERS` with a **concrete starting default of `50`** (integer). Rationale: on the ListenBrainz user base a release-group with >= 50 distinct listeners is a defensible "notable" cut and is expected to land the surviving album count in the tens of thousands; it is a starting value, not a guess left blank. It MUST then be re-tuned against the real dump so the surviving count lands in the target band ~20,000-100,000 (success criterion 1 says "not millions"): the operator runs materialize, reads the printed count, and adjusts the constant. The join keeps only release-groups with a popularity row AND listener_count >= NOTABILITY_MIN_LISTENERS AND primary_type = Album. Record the raw listener_count into entities.notability_score so the floor is auditable and re-tunable later. Document the re-tune loop in README.md.
</notability_floor>
