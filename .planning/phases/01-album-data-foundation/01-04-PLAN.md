<objective>
Give every materialized album a cover: populate the `cover_url` column (reserved in Plan 01, rows written in Plan 03) with a Cover Art Archive front-cover pointer constructed from the album's release-group MBID. Store only the URL string as a pointer; never fetch or copy image bytes.

Purpose: DATA-SOURCES.md locks "copyrighted assets are pointers, never copied files," and the Cover Art Archive is the album-cover source. A constructed template URL is a stable, zero-cost pointer that needs no per-row API call. Cover availability (some release-groups have no art) and the missing-art fallback are display concerns for Phase 2, not this phase.
Output: `pipeline/covers.py`, `pipeline/test_covers.py`.
</objective>

<cover_decision>
Cover pointers are constructed by-template, not verified per-row:
- Template: `https://coverartarchive.org/release-group/{mbid}/front-500`, where `{mbid}` is the album's release-group MBID (the same MBID stored in `entities.mbid`). `front-500` is the 500px front-cover thumbnail redirect the Cover Art Archive serves.
- Store the constructed URL string in `entities.cover_url`. Do NOT issue an HTTP request per album to verify existence, and do NOT download or store any image bytes. Verifying tens of thousands of URLs would be slow, rate-limited, and pointless: the store holds pointers, the browser resolves them at display time.
- Coverage reality: not every release-group has front cover art. A release-group with no art makes the CAA URL 404 at fetch time. Handling that gracefully (placeholder / fallback image) is a Phase 2 display concern and is explicitly out of scope here. This phase guarantees a well-formed pointer for every album, not that every pointer resolves.
- Licensing: Cover Art Archive is served under terms that permit referencing/hotlinking the image URLs; we store the reference, never the file (DATA-SOURCES.md).
</cover_decision>
