<objective>
Acquire the CC0 bulk dumps and stream the release-group-relevant tables into SQLite staging tables: MusicBrainz for album identity (release-group MBID, title, primary type, first-release year, primary artist name + MBID) and ListenBrainz for the popularity signal that Plan 03 uses as the notability floor. No live APIs for stored data (DATA-SOURCES.md architecture rule), and no full Postgres load — only the specific MB table dump files are streamed.

Purpose: Staging is the raw landing zone. Keeping the multi-million-row MB tables in SQLite staging (not a Postgres install) preserves the zero-infra, stdlib-first posture while still letting Plan 03 do a set-based join to produce the notability-floored universe.
Output: `pipeline/ingest_musicbrainz.py`, `pipeline/ingest_listenbrainz.py`, `pipeline/staging.sql`, deterministic fixtures, and `pipeline/test_ingest.py`.
</objective>

<dump_sources>
MusicBrainz full-export table dumps (CC0 core data), latest export directory:
  https://data.metabrainz.org/pub/musicbrainz/data/fullexport/latest/
The core dump `mbdump.tar.bz2` unpacks to `mbdump/<table>` files: tab-separated, NO header row, `\N` as NULL, column order matching the Postgres CREATE TABLE order.
Tables needed to build an album row (release-group scope only — do NOT load `release`, edits, or cover_art tables):
  - release_group          (id, gid[=MBID], name[=title], artist_credit, type[=primary type id], ...)
  - release_group_meta     (id[=release_group.id], ..., first_release_date_year, ...)
  - artist_credit_name     (artist_credit, position, artist, name, join_phrase)  -> take position 0 as primary
  - artist                 (id, gid[=MBID], name, ...)
Primary-type filter: release_group.type = 1 is "Album" (release_group_primary_type static list; id 1 = Album). Confirm the id-1=Album mapping against the current release_group_primary_type dump/table before hardcoding.
Various Artists is a real MB artist (gid 89ad4ac3-39f7-470e-963a-56509c546377); VA/compilation release-groups keep that as primary artist name — handle, do not drop.

ListenBrainz release-group popularity dataset (Open, MetaBrainz), keyed by release-group MBID with listen and listener counts:
  https://datasets.listenbrainz.org/  (popularity dumps; release-group popularity)
Model the parsed record as {release_group_mbid, total_listen_count, total_listener_count}. Accept JSON-lines; if the current dump ships parquet, convert to JSONL first with a documented one-liner and parse the JSONL (keep the parser stdlib-only).

Politeness: dumps are static files; a simple resumable download is fine. If any live fetch is used for validation only, User-Agent must be `TasteTest/0.1 (keith@totalemphasis.com)` and rate-limited (see poc/client.py precedent).
</dump_sources>

<column_order_warning>
Do NOT hardcode column indices from memory. Derive each table's column order from the current MusicBrainz schema (the CREATE TABLE definitions in the musicbrainz-server admin SQL, or the `mbdump/TableList`/schema shipped alongside the dump) and pin the indices you use as named constants at the top of ingest_musicbrainz.py, with a fixture-backed assertion test so a schema drift fails loudly.
</column_order_warning>
</context>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Tampering | staging inserts | mitigate | Parameterized executemany only; dump values never string-formatted into SQL |
| T-01-05 | Denial of Service | oversized/malformed dump lines | mitigate | Stream line-by-line (bounded memory), skip+count malformed lines, batch inserts; no whole-file read |
| T-01-06 | Elevation of Privilege | dump-dir / file path handling | mitigate | Treat paths as data (open with explicit encoding, no shell); no `os.system`/shell interpolation of paths |
| T-01-SC | Tampering | fetching dumps over the network | accept | CC0 static dumps from the official MetaBrainz host over HTTPS; no package-manager installs introduced in this plan |
</threat_model>
