<objective>
Stand up the serving store and its schema: a single SQLite database with a polymorphic `entities` table keyed by `(entity_type, mbid)`, a generic mechanism-tagged `atoms` table, a `sessions` table for anonymous grouping, and the indexes the ranking loop will query on. This is the foundation every later plan writes into.

Purpose: The polymorphic entity schema and generic atom table are cheap now and expensive to retrofit (PROJECT.md Key Decisions). Getting the shape right unblocks all ingestion and the entire Phase 2 pick loop.
Output: `pipeline/schema.sql`, `pipeline/db.py`, `pipeline/test_schema.py`, an initialized-on-demand `data/tastetest.db` (gitignored).
</objective>

<store_decision>
The serving store is SQLite via the Python stdlib `sqlite3` module. No third-party DB, no Postgres.
Rationale:
- Final universe is tens of thousands of album rows; SQLite handles this and the multi-million-row staging joins (Plan 02/03) comfortably.
- Python 3 stdlib is this repo's tooling norm (see CLAUDE.md Architecture); `sqlite3` is stdlib, zero infra, zero new runtime dependency.
- A single file store is trivially queryable, backup-able, and ships anywhere the Phase 2 service runs.
Note: The zero-dependency / `file://` hard rule in CLAUDE.md applies ONLY to the legacy calibration tool at the repo root, NOT to this pipeline. This pipeline may use Python packages if needed, but stdlib is preferred and sufficient here.
Store file path: `data/tastetest.db` (created on demand, gitignored).
</store_decision>

    Create `pipeline/__init__.py` (empty package marker) and `pipeline/schema.sql` containing idempotent DDL (`CREATE TABLE IF NOT EXISTS`).

    Table `entities` (SCHEMA-01, polymorphic): columns `entity_type TEXT NOT NULL`, `mbid TEXT NOT NULL`, `title TEXT NOT NULL`, `primary_artist_name TEXT`, `primary_artist_mbid TEXT`, `release_year INTEGER`, `cover_url TEXT`, `notability_score INTEGER`, `created_at INTEGER NOT NULL`, `updated_at INTEGER NOT NULL`. Primary key is the composite `(entity_type, mbid)`. `entity_type` for this phase is always the literal `'album'`, but the column and composite PK are what let a `'song'` or `'artist'` row be added later with no DDL change. Include `cover_url` now (populated by Plan 04) so no ALTER is needed later.

    Table `atoms` (SCHEMA-02, generic pairwise, entity-type agnostic): columns `id INTEGER PRIMARY KEY AUTOINCREMENT`, `entity_a TEXT NOT NULL`, `entity_b TEXT NOT NULL`, `winner TEXT NOT NULL`, `mechanism TEXT NOT NULL`, `session_id TEXT NOT NULL`, `created_at INTEGER NOT NULL`. `entity_a`, `entity_b`, and `winner` hold MBIDs (winner is the MBID of the chosen entity). No foreign key to `entities` and no entity_type column constraint, so the table is agnostic to which entity type is being compared and can accept any mechanism value. This phase writes `mechanism = 'this_or_that'`.

    Table `sessions` (SCHEMA-03, anonymous grouping): columns `session_id TEXT PRIMARY KEY`, `created_at INTEGER NOT NULL`, `last_seen_at INTEGER`. No user/account/email column exists anywhere; a session is the only grouping key for an anonymous player's picks.

    Indexes: `idx_entities_type` on `entities(entity_type)`; `idx_entities_notability` on `entities(entity_type, notability_score)`; `idx_atoms_session` on `atoms(session_id)`; `idx_atoms_mechanism` on `atoms(mechanism)`.
  </action>
