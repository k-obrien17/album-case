Intended module contract (define in types.ts, implement in insertion.ts):
  type RankingState = {
    ranked: Album[];                                  // current sorted list, index 0 = most preferred
    pending: { album: Album; lo: number; hi: number } | null;  // in-progress binary insertion
  }
  type Comparison = { candidate: Album; opponent: Album };      // the pair to show
  startPlacement(state, album): RankingState                    // begin inserting `album` (sets lo=0, hi=ranked.length)
  nextComparison(state): Comparison | null                       // null when no pending placement is active
  applyPick(state, winnerMbid): RankingState                     // narrow bounds; finalize insert when lo>=hi
</interfaces>
</context>

  <behavior>
    - Empty list: startPlacement on an empty `ranked` finalizes immediately (album becomes the sole element); nextComparison returns null after finalize.
    - First real comparison: with a non-empty list, nextComparison returns the candidate vs the element at the midpoint index `floor((lo+hi)/2)` — deterministic, never random.
    - Narrowing: if the candidate wins, hi := mid (it ranks above the opponent); if the opponent wins, lo := mid+1 (candidate ranks below). Repeat until lo>=hi, then splice the candidate into `ranked` at index lo.
    - Comparison count: inserting into a list of length k takes at most ceil(log2(k+1)) comparisons.
    - Transitivity / no self-contradiction: after inserting any sequence of albums, `ranked` is fully ordered and consistent with every pick made during each insertion — there exists no pair whose recorded picks contradict their final relative order. A property test over randomized-but-consistent preferences (a hidden total order the "player" answers by) must always reproduce that exact order.
    - Resumability: a RankingState with an active `pending` placement round-trips through JSON.stringify/parse and continues from the same lo/hi bounds (supports refresh persistence in plan 02-03).
    - Insertion-driven pairing: the opponent returned by nextComparison is always an element already in `ranked` at the computed midpoint, never an arbitrary/unrelated album (RANK-03).
  </behavior>
