# Derived album scores + keithrobrien.com music export

## Problem

`keithrobrien.com` already has a `content/collect/music.json` stub waiting: `{ "note": "All-time favorite music (hand-curated)... Empty until Keith supplies an all-time list.", "all_time": [] }`. There's a proven pattern for the equivalent movies page: `media-library/scripts/export-collect-watching.mjs` reads that app's own database read-only and writes a static `watching.json`/`watching-annual.json` into the `keithrobrien` repo, which renders it at build time with no runtime coupling. Album Case has no equivalent yet ŌĆö and unlike movies, where a score is a number someone typed in per film, Album Case's ranked list already carries more precise information than a score: an exact ordinal position, earned through real pairwise comparisons.

## Scope

In scope: a score formula computed only at export time, and a new script, `web/scripts/export-collect-music.mjs`, that reads the owner's ranking snapshot from Turso and writes a static `music.json` into the `keithrobrien` repo, mirroring `watching.json`'s shape and conventions.

Out of scope: any change to Album Case's own schema, data model, or UI ŌĆö no score is ever stored or displayed inside the app itself; a per-year/per-decade cut (deferred, see Non-goals); the already-approved bulk-album-list-import spec (`2026-07-11-bulk-album-list-import-design.md`) ŌĆö unaffected, since imported albums flow through the same ranked-list mechanism as everything else and pick up a derived score automatically once genuinely placed.

## Why derived, not stored (the core decision)

Media-library's `score` is a genuinely independent number a person typed in ŌĆö appropriate there, since that product has no ordinal ranking model at all. Album Case's own core rule, from `web/src/ranking/insertion.ts`'s doc comment, is the opposite: *"transitive by construction... NOT Elo. There are no ratings, scores, or probabilistic outcomes here, only ordinal position."* Storing a score field on albums would be exactly the class of model that rule exists to rule out.

Resolution: score is computed only at export time, from rank position, in the export script ŌĆö never persisted in Turso, never added to any API response, never shown anywhere in the app's own UI. Album Case's core stays completely untouched; the score only exists as a number printed into a JSON file for a different website to read.

## Score formula

```
raw(rank, total) = 1 + 9 * (total - rank) / (total - 1)
score(rank, total) = Math.round(raw(rank, total) * 100) / 100   // 2 decimal places
```

- Rank 1 (best) ŌåÆ `10.00`; rank `total` (worst) ŌåÆ `1.00`.
- **Two decimal places, not one.** Verified empirically before locking this in: at 1 decimal (matching movies), 244 ranked albums collapse to 91 distinct values, worst case 3 albums sharing one number ŌĆö including a tie between rank #1 and rank #2. At 2 decimals: 244 distinct values, zero ties. Movies use 1 decimal because it's a hand-typed human number; Album Case's score is formula-derived from an exact position, so the extra precision is honest, not fake.
- Score is a strictly monotonic function of rank, so "top N by score" and "top N by rank" are the same set of albums ŌĆö the formula only supplies the display number, never the selection.

## Export script

`web/scripts/export-collect-music.mjs`, modeled directly on `media-library/scripts/export-collect-watching.mjs`:

- Read-only against the same Turso database the app already uses (`TURSO_DATABASE_URL` / `TURSO_AUTH_TOKEN`), queried directly ŌĆö no HTTP round-trip through the deployed API, same as how the movies script talks straight to its SQLite file rather than calling its own app's endpoints.
- Fetches the owner's current ranking snapshot (the `ranked` array, already in exact order ŌĆö same table/shape `/api/ranking` already serves).
- Takes the top N (default 10, matching movies' `top_movies` cut; overridable via a `TOP_N` env var).
- Maps each album to:
  ```json
  { "title": "...", "artist": "...", "year": 2000, "score": 10.0, "type": "album", "mb_url": "https://musicbrainz.org/release-group/<mbid>" }
  ```
  `artist` = `primary_artist_name`, `year` = `release_year`, `mb_url` plays the same role `tmdb_url` does for movies (a canonical external link).
- Output path defaults to the sibling `../../keithrobrien/content/collect/music.json`, overridable via `COLLECT_OUT` ŌĆö same convention as the movies script.
- Full payload:
  ```json
  {
    "generated_at": "2026-07-11",
    "source": "album-case",
    "note": "Generated by album-case/web/scripts/export-collect-music.mjs. Do not hand-edit.",
    "top_albums": [ /* ... */ ]
  }
  ```
  This replaces the current stub's `{ note, all_time: [] }` shape outright ŌĆö the new `note` field documents the change so nobody mistakes it for still being hand-editable.

## Running it

Manual, on-demand ŌĆö same as the movies script: no cron, no CI hook, no git commit hook. Keith runs `node web/scripts/export-collect-music.mjs` after a ranking session, reviews the diff in the `keithrobrien` repo, and commits it there himself.

## Error handling

- Fewer than N albums ranked: export whatever exists, no error. (No `min_pool` gate like `watching-annual.json`'s per-year cuts ŌĆö that gate exists there because a thin per-year bucket looks broken; a single all-time top-N list doesn't have that failure mode.)
- Missing Turso env vars: fail loudly to stderr with a non-zero exit, matching the existing pipeline scripts' convention ŌĆö never silently write an empty or stale file over a real one.
- `keithrobrien` repo not present at the expected sibling path: fail with a clear message pointing at `COLLECT_OUT`, same guard the movies script already has.

## Testing

Media-library's own `export-collect-watching.mjs` has no test file ŌĆö this class of script isn't part of that project's test surface, and Album Case's own convention (`CLAUDE.md`: tests only when asked) matches. The score formula was validated by hand before being locked in (see the tie-count comparison above); no colocated test is planned unless requested.

## Non-goals

- Per-year/per-decade cuts (`watching-annual.json`'s equivalent) ŌĆö deferred until a flat top-N list feels too thin to be interesting.
- Any score field, column, or UI surface inside Album Case itself.
- Automatic/scheduled regeneration ŌĆö stays a manual, reviewed step, same as movies.
- Any change to the bulk-album-list-import spec.
